# Grafana Loki 구조적 개선 제안서

---

## 1. 프로젝트 개요

Grafana Loki는 Prometheus에서 영감을 받은 수평 확장 가능한 멀티테넌트 로그 집계 시스템이다. 로그 내용을 인덱싱하지 않고 레이블 셋만 인덱싱하는 설계 철학을 기반으로, 압축된 비정형 로그를 오브젝트 스토리지에 저장하여 비용 효율성을 극대화한다.

핵심 컴포넌트는 Distributor(로그 수신/검증/라우팅), Ingester(인메모리 버퍼링/WAL/플러시), Querier(듀얼 소스 쿼리 실행), Query Frontend(쿼리 분할/샤딩/캐싱), Compactor(인덱스 압축/보존 정책), Bloom Gateway(블룸 필터 기반 쿼리 가속)로 구성된다. 모듈 시스템(`pkg/loki/modules.go`)을 통해 모노리스, Simple Scalable(read/write/backend), 마이크로서비스 등 다양한 배포 모드를 지원한다.

본 문서는 Loki의 아키텍처를 심층 분석하여 구조적 한계점과 성능 병목을 식별하고, 구체적인 개선 방안을 제시한다.

---

## 2. 현재 구조적 한계점 및 성능 병목

### 2.1 Ingester 플러시 시스템의 구조적 비효율

**현상**: Ingester의 플러시 시스템(`pkg/ingester/flush.go`)은 `FlushCheckPeriod`(기본 30초)마다 `sweepUsers()` -> `sweepInstance()` -> `sweepStream()` 순으로 전체 스트림을 순회하며 플러시 대상을 결정한다. 이 과정에서 모든 테넌트의 모든 스트림을 매 주기마다 전수 검사하며, 플러시 큐에 삽입 후 `flushRateLimiter`로 속도를 제한한다.

**병목 지점**:
- `sweepStream()`에서 `stream.chunkMtx.RLock()`을 획득하여 각 스트림의 청크 상태를 확인하는데, 대규모 환경(수백만 스트림)에서 락 경합이 발생한다.
- 플러시 큐 인덱스를 `uint64(stream.fp) % uint64(ConcurrentFlushes)`로 결정하는데, 핑거프린트 분포가 균등하지 않을 경우 특정 큐에 부하가 집중된다.
- `setFlushRate()`에서 전체 큐 길이를 합산하여 `flushRateLimiter`를 동적 조정하지만, 이 계산이 FlushCheckPeriod 단위로만 갱신되어 버스트 상황에 대한 반응이 느리다.
- 플러시 실패 시 `flushBackoff`만큼 지연 후 재큐잉되는데, 오브젝트 스토어 장애가 지속되면 큐가 무한히 증가할 수 있다.

**영향**: 대규모 멀티테넌트 환경에서 플러시 지연으로 인한 메모리 증가, 오브젝트 스토어에 대한 불균등한 I/O 패턴 발생.

### 2.2 Querier 듀얼 소스 병합의 중복 데이터 오버헤드

**현상**: `pkg/querier/querier.go`의 `SelectLogs()`/`SelectSamples()`는 Ingester와 Store를 동시에 쿼리하고 `MergeEntryIterator`(`pkg/iter/entry_iterator.go`)로 병합한다. `buildQueryIntervals()`에서 시간 분할을 시도하지만, `IngesterQueryStoreMaxLookback`이 미설정(기본값 0)인 경우 양쪽 소스에 동일한 전체 시간 범위를 전송한다.

**병목 지점**:
- 겹치는 시간 구간에서 동일 데이터가 Ingester와 Store 양쪽에서 반환되며, `mergeEntryIterator.fillBuffer()`에서 `streamHash + Timestamp + Line` 기반 중복 제거를 수행한다. 이 과정에서 O(N) 비교가 발생하고, 동일 타임스탬프의 엔트리가 많을수록 성능이 급격히 저하된다.
- Loser Tree 기반 K-way merge는 O(log K) 복잡도로 효율적이지만, Ingester가 여러 개인 경우(replication factor 3) 동일 데이터가 3배로 반환되어 K가 불필요하게 증가한다.
- `SampleIteratorHeap`(`pkg/iter/sample_iterator.go`)은 `container/heap`을 사용하는데, EntryIterator의 Loser Tree와 달리 O(log K)이지만 캐시 지역성이 떨어진다.

**영향**: 오브젝트 스토리지 I/O 및 네트워크 대역폭 낭비, Querier 메모리 사용량 증가, 쿼리 지연 시간 증가.

### 2.3 Compactor 단일 인스턴스 병목

**현상**: `pkg/compactor/compactor.go`의 Compactor는 Ring 기반 리더 선출(`ringReplicationFactor=1`, `ringNumTokens=1`)을 통해 클러스터 내 단 하나의 인스턴스만 인덱스 압축과 보존 정책을 수행한다. `loop()`에서 5초마다 `ring.Get(ringKeyOfLeader, ring.Write)`로 리더 확인 후, 리더인 경우에만 `tablesManager.start()`를 호출한다.

**병목 지점**:
- 단일 인스턴스가 모든 테이블의 압축, 보존 정책 적용, 삭제 처리를 담당하므로 수직 확장 한계가 존재한다. `MaxCompactionParallelism`(기본 1)을 높여도 단일 인스턴스의 CPU/메모리/디스크 I/O에 제한된다.
- `tablesManager.start()`에서 Ring 안정화를 위해 `CompactionInterval`(기본 10분)만큼 초기 대기하므로, 리더 전환 시 최소 10분의 압축 공백이 발생한다.
- `SortTablesByRange()`가 최신 테이블 우선 정렬을 수행하지만, 오래된 테이블의 압축이 지연되면 인덱스 파일 수가 누적되어 쿼리 성능이 저하된다.
- 수평 스케일링 모드(`HorizontalScalingModeMain/Worker`)가 존재하지만 삭제 작업에만 적용되고, 핵심인 인덱스 압축 자체는 여전히 단일 인스턴스에서 수행된다.

**영향**: 대규모 클러스터에서 인덱스 압축 지연, 보존 정책 적용 지연, 쿼리 시 다수의 미압축 인덱스 파일 스캔으로 인한 성능 저하.

### 2.4 Distributor의 스트림 샤딩과 레이트 리밋 간의 정합성 문제

**현상**: `pkg/distributor/distributor.go`의 `PushWithResolver()`에서 스트림별 검증, 레이트 리밋, 스트림 샤딩이 순차적으로 수행된다. 스트림 샤딩(`maybeShardStreams`)은 단일 고카디널리티 스트림을 여러 개의 작은 스트림으로 분할하여 Ingester 부하를 분산하는데, 이 과정에서 원본 스트림의 레이블에 `__loki_shard__` 레이블이 추가되어 해시값이 변경된다.

**병목 지점**:
- 글로벌 레이트 리밋(`ingestionRateLimiter.AllowN`)은 샤딩 이전의 `totalEntriesSize`로 계산되지만, 실제 Ingester에 도달하는 데이터는 샤딩 후 메타데이터 오버헤드가 추가된 크기이다. 이로 인해 실제 인제스트 속도가 설정된 리밋을 초과할 수 있다.
- `ShardTracker`가 `StreamRateCalculator`의 속도 정보를 기반으로 샤딩 결정을 내리는데, 이 정보는 Ingester에서 주기적으로 보고되므로 실시간성이 떨어진다. 급격한 트래픽 변화 시 샤딩이 과도하거나 부족할 수 있다.
- 워커 풀(`PushWorkerCount=256`)이 고정 크기이고 `ingesterTasks` 채널이 고정 버퍼인데, 대규모 버스트 상황에서 채널이 가득 차면 고루틴 블로킹이 발생하여 전체 Push 지연이 증가한다.
- `parseStreamLabels()`에서 LRU 캐시(`labelCache`)를 사용하지만, 높은 카디널리티의 동적 레이블이 많은 환경에서 캐시 미스율이 높아져 레이블 파싱 오버헤드가 증가한다.

**영향**: 레이트 리밋의 정확도 저하, 트래픽 버스트 시 Push 지연 증가, 높은 카디널리티 환경에서의 CPU 오버헤드.

### 2.5 블룸 필터 빌드/쿼리 파이프라인의 비효율

**현상**: 블룸 필터 서브시스템(`pkg/storage/bloom/v1/`, `pkg/bloomgateway/`, `pkg/bloombuild/`)은 Planner가 빌드 작업을 생성하고 Builder가 실행하는 구조이다. `ScalableBloomFilter`는 필터가 가득 차면 새 필터를 `fpRate * r^n`으로 추가하여 동적 확장한다.

**병목 지점**:
- 블룸 블록이 청크 단위로 빌드되므로, 청크 내 로그 라인 수가 크게 변동하는 환경에서 일부 블룸 필터는 과도하게 크고(false positive rate 낮지만 공간 낭비), 일부는 과도하게 작다(높은 false positive rate).
- `BloomPageHeader`의 `N`, `Offset`, `Len`, `DecompressedLen` 구조에서 페이지 단위 랜덤 I/O가 발생하며, 대규모 쿼리 시 다수의 블룸 블록을 다운로드/압축 해제해야 한다.
- Bloom Gateway의 `FilterChunkRefs`가 동기적으로 블룸 블록을 로드하여 테스트하므로, 블룸 블록이 오브젝트 스토어에만 존재하는 경우(캐시 미스) 쿼리 지연이 크게 증가한다.
- JumpHash 기반 Gateway 라우팅이 시리즈 핑거프린트로만 분배하여, 특정 시리즈에 대량의 청크가 존재할 경우 해당 Gateway에 부하가 집중된다.

**영향**: 블룸 필터 빌드 비용 대비 쿼리 가속 효과가 일정하지 않음, 캐시 미스 시 쿼리 지연 증가.

## 3. 개선 제안

### 3.1 Ingester 플러시 시스템의 적응형 재설계

**문제 상세**:
Ingester의 플러시 파이프라인은 `loop()` -> `sweepUsers()` -> `sweepInstance()` -> `sweepStream()` -> `shouldFlushChunk()` 순으로 전수 검사 방식을 사용한다. 스트림 수가 수백만 개에 달하는 대규모 환경에서, 매 30초마다 모든 스트림을 순회하는 것은 불필요한 CPU 사이클 소모와 락 경합을 초래한다.

**근본 원인**:
플러시 대상 식별이 "폴링(polling)" 방식으로 구현되어 있으며, 스트림/청크 상태 변경 시 즉시 플러시 큐에 삽입하는 "이벤트 드리븐" 방식이 아니다. `chunkDesc`의 `lastUpdated`와 `closed` 상태가 변경될 때 플러시 결정을 내리지 않고, 주기적 스위프에서만 결정한다.

**제안 솔루션**:

1. **이벤트 기반 플러시 트리거**: `stream.Push()`에서 청크가 `closed` 상태로 전환되거나, `lastUpdated` 이후 `MaxChunkIdle` 이상 경과한 경우 즉시 플러시 큐에 삽입하는 방식으로 변경한다. 타이머 휠(Timer Wheel) 자료구조를 도입하여 idle 타임아웃을 O(1)로 관리한다.

2. **적응형 플러시 레이트**: 현재의 `setFlushRate()`는 전체 큐 길이 기반의 단순 계산이다. 오브젝트 스토어의 응답 지연을 피드백으로 활용하는 PID 컨트롤러 기반 적응형 레이트 제어를 도입한다. `flushOp()` 실행 시간을 지수 이동 평균으로 추적하고, 응답 지연이 증가하면 플러시 속도를 자동 감소시킨다.

3. **플러시 큐 리밸런싱**: 핑거프린트 기반 큐 할당 대신, 각 큐의 현재 길이를 고려한 최소 부하 큐 선택(Least-Loaded Queue) 방식을 도입한다.

**예상 효과**:
- 플러시 지연 시간 50-70% 감소 (이벤트 드리븐 방식으로 대기 시간 제거)
- 전수 검사 CPU 오버헤드 80% 이상 감소
- 오브젝트 스토어 장애 시 메모리 증가 속도를 PID 제어로 완화

**구현 난이도**: 중상 (Timer Wheel 및 PID 컨트롤러 구현 필요, 기존 플러시 로직의 상당 부분 리팩토링)

### 3.2 Querier 듀얼 소스 쿼리의 지능형 시간 분할

**문제 상세**:
`buildQueryIntervals()`의 현재 로직은 `IngesterQueryStoreMaxLookback`이 0(기본값)이면 Ingester와 Store 양쪽에 동일한 전체 시간 범위를 전송한다. 이로 인해 겹치는 구간에서 데이터가 중복 반환되고, `mergeEntryIterator.fillBuffer()`에서 중복 제거 비용이 발생한다.

**근본 원인**:
Ingester가 마지막으로 플러시한 시점(flush watermark)을 Querier가 정확히 알 수 없다. Ingester의 `RetainPeriod`(기본 0) 설정에 따라 플러시 후에도 메모리에 보관되는 기간이 다르며, 각 스트림의 플러시 상태가 개별적이므로 단일 시간 경계로 분할하기 어렵다.

**제안 솔루션**:

1. **Ingester Flush Watermark 노출**: 각 Ingester가 gRPC 메타데이터로 "가장 오래된 미플러시 청크의 타임스탬프"를 노출한다. Querier가 이 정보를 활용하여 Store 쿼리의 시간 범위를 `[queryStart, min(watermarks)]`로 제한한다.

2. **적응형 시간 분할**: `IngesterQueryStoreMaxLookback`을 런타임에 자동 계산하는 모드를 추가한다. Ingester의 `MaxChunkAge`(기본 2시간) + `FlushCheckPeriod`(30초) + 여유분을 기반으로 안전한 겹침 구간을 자동 설정한다.

3. **중복 제거 최적화**: `fillBuffer()`의 중복 제거에서 동일 `streamHash`의 엔트리를 사전 그룹화하고, 해시 기반 O(1) 중복 검사로 전환한다. 현재의 선형 비교 대신 `xxhash(Line)`을 사전 계산하여 비교한다.

4. **SampleIterator Loser Tree 통합**: `SampleIteratorHeap`을 `container/heap` 대신 EntryIterator와 동일한 Loser Tree 기반으로 통합하여 캐시 지역성을 개선한다.

**예상 효과**:
- 중복 데이터 전송량 70-90% 감소 (flush watermark 기반 분할)
- 중복 제거 CPU 비용 60% 감소 (해시 기반 O(1) 비교)
- 메트릭 쿼리(SelectSamples) 성능 15-20% 향상 (Loser Tree 통합)

**구현 난이도**: 중 (Ingester gRPC 인터페이스 확장 필요, 기존 Iterator 구현 리팩토링)

### 3.3 Compactor 분산 압축 아키텍처

**문제 상세**:
Compactor가 Ring 기반 단일 인스턴스로 동작하여 수직 확장 한계가 존재한다. `tablesManager.runCompaction()`에서 테이블 목록을 순회하며 `CompactTable()`을 호출하는데, `MaxCompactionParallelism`을 높여도 단일 인스턴스의 I/O 대역폭에 제한된다. 리더 전환 시 `CompactionInterval`만큼의 공백은 인덱스 압축 SLA를 위반할 수 있다.

**근본 원인**:
인덱스 압축의 원자성 보장을 위해 단일 인스턴스 실행이 필요하다는 전제가 있지만, 이는 테이블 단위 잠금으로 해결 가능하다. 현재 `tableLocker`가 테이블 수준 잠금을 이미 제공하고 있으므로, 이를 분산 잠금으로 확장하면 된다.

**제안 솔루션**:

1. **테이블 단위 분산 잠금**: 현재의 로컬 `tableLocker`를 분산 잠금(오브젝트 스토어 기반 리스 또는 etcd/consul 기반)으로 교체한다. 각 Compactor 인스턴스가 테이블 단위로 잠금을 획득하고 독립적으로 압축을 수행한다.

2. **테이블-인스턴스 매핑 Ring**: Compactor용 Ring에서 `ringNumTokens`을 테이블 수에 비례하여 증가시키고, 각 테이블의 이름 해시를 Ring에서 조회하여 담당 인스턴스를 결정한다. 이를 통해 테이블이 자동으로 인스턴스 간에 분배된다.

3. **무중단 리더 전환**: 리더 전환 시 `CompactionInterval` 대기를 제거하고, 테이블 단위 잠금으로 동시 접근을 방지한다. 새 리더는 즉시 미완료 테이블부터 압축을 시작한다.

4. **보존 정책 병렬화**: `RetentionEnabled` 시 Mark-Sweep 2단계에서 Mark 단계를 테이블 단위로 병렬화한다. 현재는 `applyRetention=true`인 별도 루프에서 순차 실행되지만, 테이블 단위 독립성을 활용하여 여러 인스턴스에서 동시에 Mark를 수행한다.

**예상 효과**:
- 압축 처리량 N배 증가 (N = Compactor 인스턴스 수)
- 리더 전환 공백 시간 제거 (10분 -> 0)
- 보존 정책 적용 지연 50-80% 감소

**구현 난이도**: 상 (분산 잠금 구현, Ring 토큰 전략 변경, 기존 단일 인스턴스 가정 코드 전면 수정)

### 3.4 Distributor 레이트 리밋/샤딩 일관성 개선

**문제 상세**:
스트림 샤딩 이후 메타데이터 오버헤드(`__loki_shard__` 레이블)가 추가되어 실제 인제스트 크기가 글로벌 레이트 리밋 계산 시점의 `totalEntriesSize`보다 크다. 또한 `ShardTracker`의 스트림 속도 정보가 `OwnedStreamsCheckInterval`(30초) 주기로 갱신되어 실시간 트래픽 변화를 반영하지 못한다.

**근본 원인**:
레이트 리밋 검사(단계 7)와 스트림 샤딩(단계 6k)이 순서적으로 분리되어 있으며, 샤딩으로 인한 크기 변화를 레이트 리밋에 피드백하는 메커니즘이 없다. 또한 글로벌 레이트 리밋이 `healthyInstancesCount` 기반으로 분산 계산되는데, Distributor 인스턴스 간 요청 분포가 균등하지 않을 수 있다.

**제안 솔루션**:

1. **샤딩 후 레이트 리밋**: 레이트 리밋 검사를 스트림 샤딩 이후로 이동하고, 샤딩된 스트림의 실제 크기로 리밋을 적용한다. 현재 코드에서 단계 6k와 단계 7의 순서를 교환하되, 샤딩 비용을 미리 계산하여 정확한 크기 추정치를 사용한다.

2. **실시간 스트림 속도 추적**: `StreamRateCalculator`의 갱신 주기를 줄이는 대신, Distributor 내부에서 Push 요청 시 로컬 속도를 실시간 추적하는 슬라이딩 윈도우 카운터를 추가한다. Ingester 보고 속도와 로컬 속도를 가중 평균하여 샤딩 결정에 활용한다.

3. **적응형 워커 풀**: 고정 크기 `PushWorkerCount`(256) 대신, `ingesterTasks` 채널 사용률을 모니터링하여 워커 수를 동적으로 조절하는 적응형 풀을 구현한다. 채널 점유율이 80%를 초과하면 워커를 추가하고, 20% 미만이면 감소시킨다.

4. **레이블 캐시 개선**: 현재 LRU 캐시의 크기가 고정이며, 높은 카디널리티 환경에서 미스율이 높다. 테넌트별 적응형 캐시 크기 조절과, 캐시 미스 시 오프셋 기반 부분 매칭(레이블 셋의 접두사가 동일한 경우 부분 재사용)을 도입한다.

**예상 효과**:
- 레이트 리밋 정확도 향상 (실제 인제스트 크기 기반)
- 트래픽 버스트 시 Push 지연 40-60% 감소 (적응형 워커 풀)
- 높은 카디널리티 환경에서 레이블 파싱 CPU 30% 감소

**구현 난이도**: 중 (코드 순서 변경은 단순하지만, 적응형 워커 풀과 캐시 개선은 주의 깊은 구현 필요)

### 3.5 블룸 필터 빌드/쿼리 파이프라인의 적응형 최적화

**문제 상세**:
`ScalableBloomFilter`의 확장 패턴(`fpRate * r^n`, `capacity * s^n`)이 모든 청크에 동일하게 적용되어, 로그 라인 수 변동이 큰 환경에서 공간 효율이 떨어진다. Bloom Gateway의 `FilterChunkRefs`는 블룸 블록을 동기적으로 로드하여 캐시 미스 시 쿼리 지연이 크게 증가한다.

**근본 원인**:
블룸 필터의 크기가 청크 생성 시점에 결정되며, 실제 쿼리 패턴(어떤 필터 표현식이 자주 사용되는지)을 반영하지 않는다. 또한 Bloom Gateway가 블룸 블록을 "요청 시 로드"하는 pull 방식이므로, 첫 번째 쿼리에서 항상 콜드 스타트가 발생한다.

**제안 솔루션**:

1. **쿼리 패턴 기반 적응형 블룸 빌드**: Bloom Planner에 쿼리 통계 수집 모듈을 추가하여, 자주 사용되는 필터 표현식(예: `|= "error"`, `|= "timeout"`)의 빈도를 추적한다. 자주 쿼리되는 시리즈/시간 범위의 블룸 필터는 낮은 FPR(0.001)로, 드물게 쿼리되는 영역은 높은 FPR(0.05)로 빌드하여 전체 스토리지 비용을 최적화한다.

2. **블룸 블록 프리페치**: Bloom Gateway 시작 시 최근 N일(설정 가능)의 블룸 블록을 비동기로 프리로드한다. `PrefetchBloomBlocks` 로직을 확장하여, 쿼리 패턴 통계에 기반한 예측적 프리페치를 구현한다.

3. **계층적 블룸 필터**: 청크 단위 블룸 필터 외에, 시리즈 단위 상위 블룸 필터를 추가한다. 쿼리 시 먼저 시리즈 블룸 필터로 전체 시리즈를 빠르게 제거하고, 남은 시리즈의 청크 블룸 필터만 로드한다. 이로써 블룸 블록 로드 횟수를 줄인다.

4. **JumpHash 개선**: 시리즈 핑거프린트 외에 청크 수를 가중치로 고려하는 가중 일관성 해싱으로 전환하여 Gateway 간 부하를 균등화한다.

**예상 효과**:
- 블룸 필터 스토리지 비용 30-40% 감소 (적응형 FPR)
- 캐시 콜드 스타트 시 쿼리 지연 60-80% 감소 (프리페치)
- 대규모 쿼리 시 블룸 블록 로드 횟수 50% 감소 (계층적 필터)

**구현 난이도**: 상 (쿼리 통계 수집 인프라 구축, 계층적 블룸 필터 설계 및 하위 호환성 보장)

## 5. 기대 성과

### 정량적 성과

| 지표 | 현재 상태 (추정) | 개선 후 목표 | 개선 폭 |
|------|-----------------|-------------|---------|
| 쿼리 P99 지연 시간 (30일 범위) | ~15초 | ~5초 | **67% 감소** |
| Ingester 플러시 지연 (P95) | ~45초 | ~10초 | **78% 감소** |
| 중복 데이터 전송량 | ~30% 오버헤드 | ~5% 이하 | **83% 감소** |
| Compactor 압축 완료 시간 (1000 테이블) | ~4시간 | ~1시간 | **75% 감소** |
| 블룸 필터 스토리지 비용 | 기준 100% | ~65% | **35% 감소** |
| Push P99 지연 (버스트 시) | ~500ms | ~200ms | **60% 감소** |

### 정성적 성과

1. **운영 안정성 향상**: 적응형 플러시와 PID 기반 레이트 제어로 오브젝트 스토어 장애 시 시스템의 자체 보호 능력이 강화된다. Compactor의 분산 처리로 단일 장애점이 제거된다.

2. **비용 효율성**: 중복 데이터 전송 감소로 오브젝트 스토리지 I/O 비용이 절감되고, 적응형 블룸 필터로 스토리지 비용이 최적화된다. 쿼리 성능 향상으로 동일 부하를 처리하는 데 필요한 Querier 인스턴스 수가 감소한다.

3. **확장성 개선**: Compactor의 분산 압축으로 클러스터 규모에 비례한 압축 처리량을 확보한다. Distributor의 적응형 워커 풀로 트래픽 변동에 유연하게 대응한다.

4. **개발자 경험**: 이벤트 드리븐 플러시와 flush watermark 기반 쿼리는 시스템 동작을 더 예측 가능하게 만들어, 디버깅과 성능 튜닝을 용이하게 한다.

5. **업스트림 기여 가능성**: 본 개선 사항들은 Grafana Loki 프로젝트의 기존 설계 철학과 일치하며, 업스트림 PR로 기여할 수 있는 범위 내에 있다. 특히 3.2(중복 제거 최적화)와 3.4(레이트 리밋 일관성)는 기존 이슈에서 논의되고 있는 주제이다.
