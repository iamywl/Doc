# 제2장: 아키텍처 및 설계 원칙

---

## 2.1 전체 아키텍처 개요

H-Cloud는 IaaS(Infrastructure as a Service), PaaS(Platform as a Service) 계층을 자체 구축하여 제공하는 프라이빗 클라우드 플랫폼이다[3][9]. 전체 아키텍처는 하드웨어 인프라 계층, 가상화 및 컨테이너 계층, 플랫폼 서비스 계층, 애플리케이션 계층으로 구성된다.

### 2.1.1 계층 구조

```
┌──────────────────────────────────────────────────────┐
│            애플리케이션 계층 (Application Layer)          │
│  커넥티드카 서비스 | 업무 시스템 | 스마트팩토리 | AI/ML      │
├──────────────────────────────────────────────────────┤
│            플랫폼 서비스 계층 (PaaS Layer)                │
│  HKS(Kubernetes) | CI/CD | 모니터링 | 로깅 | 보안         │
├──────────────────────────────────────────────────────┤
│         인프라 서비스 계층 (IaaS Layer)                   │
│  HCS(Compute) | H2O(Storage) | HFabric(Network)       │
│  HMetal(Bare Metal) | HLB(Load Balancer)               │
├──────────────────────────────────────────────────────┤
│         가상화/컨테이너 계층 (Virtualization Layer)        │
│  Kubernetes | Docker/Containerd | VM                    │
├──────────────────────────────────────────────────────┤
│         하드웨어 인프라 계층 (Hardware Layer)              │
│  Dell 서버 | Intel 프로세서 | Equinix 데이터센터            │
└──────────────────────────────────────────────────────┘
```

각 계층에 대한 상세 설명은 다음과 같다:

**하드웨어 인프라 계층**: Dell 하드웨어 기반의 온프레미스 서버와 Intel 프로세서를 사용하며, Equinix Metal을 통한 글로벌 데이터센터 코로케이션을 활용한다[7][6].

**가상화/컨테이너 계층**: IaaS 기반 기술로 OpenStack을 활용하며[101], 쿠버네티스(Kubernetes) 기반의 컨테이너 오케스트레이션과 가상머신(VM) 환경을 모두 지원한다[4][5].

**인프라 서비스 계층(IaaS)**: 클라우드 컴퓨트(HCS), 베어메탈(HMetal), 네트워크(HFabric), 로드밸런싱(HLB), 오브젝트 스토리지(H2O) 등 오픈소스 기반의 자체 개발 서비스를 제공한다[2][3].

**플랫폼 서비스 계층(PaaS)**: HKS(Hyundai Kubernetes Service), GitLab, Harbor, ArgoCD 기반의 CI/CD 파이프라인, Hubble 모니터링, Sonar 헬스체크 등의 플랫폼 서비스를 제공한다[4][5].

**애플리케이션 계층**: 커넥티드카 서비스, 그룹 업무 시스템, 스마트팩토리, AI/ML 워크로드 등 실제 비즈니스 애플리케이션이 구동된다[1][9].

## 2.2 아키텍처 전환: 모놀리식에서 MSA로

### 2.2.1 기존 모놀리식 아키텍처의 한계

H-Cloud 이전의 현대자동차그룹 IT 시스템은 모놀리식(Monolithic) 아키텍처를 기반으로 운영되었다[1][4]. 모놀리식 아키텍처에서는 모든 기능이 하나의 거대한 애플리케이션으로 통합되어 있어, 다음과 같은 구조적 한계가 존재하였다:

- **장애 전파**: 특정 서비스에 문제가 발생하면 전체 서비스에 영향을 미쳐 시스템 전체가 중단될 수 있었다[1].
- **배포 복잡성**: 작은 변경사항 하나를 반영하기 위해서도 전체 애플리케이션을 재배포해야 하였다.
- **확장성 제약**: 특정 기능에 대한 트래픽 증가 시 전체 시스템을 확장해야 하여 자원 효율성이 낮았다.
- **기술 종속**: 단일 기술 스택에 종속되어 새로운 기술 도입이 어려웠다.

### 2.2.2 MSA로의 전환

이러한 한계를 극복하기 위해 현대자동차그룹은 컨테이너 기반의 마이크로서비스 아키텍처(MSA, Microservices Architecture)로 전환하였다[4][5]. MSA 전환을 통해 각 기능이 독립적인 서비스 단위로 분리되었으며, 부분 장애가 발생하더라도 해당 서비스만 영향을 받고 나머지 서비스는 정상 운영되는 구조를 확보하였다[4].

MSA 전환 과정에서 현대오토에버는 VMware Tanzu Labs와 협업하여 도메인 주도 설계(DDD, Domain-Driven Design)를 도입하였다[102]. 이를 통해 배포 시간을 기존 14일에서 3일 미만으로 단축하였으며, 개별 마이크로서비스를 독립적으로 업데이트하여 제로 다운타임(Zero Downtime) 배포를 달성하였다[102]. 2020년 9월부터는 임직원 대상 MSA 교육 프로그램을 운영하여 전사적 MSA 역량 강화를 추진하고 있다[103].

MSA 전환의 핵심 원칙은 다음과 같다:

1. **서비스 독립성**: 각 마이크로서비스는 독립적으로 개발, 배포, 확장이 가능하다[4].
2. **비동기 통신**: 독립적으로 분산된 서비스 간에는 비동기 통신 방식을 채택하여, 동시 사용자 증가에 유연하게 대응할 수 있다[4].
3. **컨테이너 패키징**: 각 마이크로서비스는 독립적인 컨테이너로 패키징되어 배포된다[5].
4. **도메인 주도 설계(DDD)**: 비즈니스 도메인 중심으로 서비스 경계를 정의하여, 개발 복잡성을 낮추고 유지보수성을 향상시킨다[102].
5. **느슨한 결합(Loose Coupling)**: 서비스 간 의존성을 최소화하여 개별 서비스의 변경이 다른 서비스에 미치는 영향을 줄인다.

## 2.3 핵심 설계 원칙

H-Cloud의 설계 원칙은 다음의 5가지로 요약할 수 있다[1][2][3]:

### 2.3.1 글로벌 인프라 표준화

원활한 마이그레이션과 확장을 위해 글로벌 인프라 표준을 먼저 정의한 후, 해당 표준에 맞춰 H-Cloud 제품을 자체(In-House) 개발하는 접근 방식을 취하였다[1]. 이를 통해 어느 리전에서든 동일한 수준의 서비스를 제공할 수 있으며, 새로운 리전 확장 시에도 표준화된 절차를 따를 수 있다.

### 2.3.2 스케일아웃(Scale-out) 구조

인프라 단부터 수평적으로 확장 가능한 스케일아웃 구조로 설계되었다[1]. 이는 커넥티드카 서비스 가입자 수 증가에 따른 트래픽 증가에 탄력적으로 대응하기 위한 것이다. 스케일업(Scale-up) 방식은 물리적 한계가 있는 반면, 스케일아웃 방식은 노드를 추가하는 것만으로 처리 용량을 확장할 수 있어 대규모 서비스에 적합하다.

### 2.3.3 분산 배치

서비스 노드들이 하나의 데이터센터에 집중되지 않고, 글로벌로 분산 배치되어 있다[1][7]. 이를 통해 단일 데이터센터 장애 시에도 서비스 연속성을 보장할 수 있으며, 사용자에게 지리적으로 가까운 리전에서 서비스를 제공하여 응답 지연시간(Latency)을 최소화할 수 있다.

### 2.3.4 비동기 통신

독립적으로 분산된 서비스 간에는 비동기 통신 방식을 채택하였다[4]. 동기 통신 방식은 요청-응답 사이에 대기 시간이 발생하여 처리 효율이 낮아질 수 있는 반면, 비동기 통신 방식은 메시지 큐(Message Queue) 등을 활용하여 서비스 간 요청을 비동기적으로 처리함으로써 동시 사용자 증가 시에도 안정적인 응답 성능을 유지할 수 있다.

### 2.3.5 오픈소스 기반 자체 개발

H-Cloud의 핵심 서비스 구성요소는 모두 오픈소스를 기반으로 자체 개발되었다[2][3]. 이는 특정 벤더에 대한 종속성(Vendor Lock-in)을 방지하고, 기술적 자율성을 확보하기 위한 전략이다. 오픈소스를 내부 환경에 맞게 광범위한 커스터마이징 과정을 거쳐 적용하고 있다[1].

## 2.4 하이브리드 멀티클라우드 아키텍처

H-Cloud는 순수 프라이빗 클라우드만을 운영하는 폐쇄적 구조가 아니라, 퍼블릭 클라우드와의 연계를 통한 하이브리드 멀티클라우드 아키텍처를 구현하고 있다[6][26].

### 2.4.1 하이브리드 클라우드 구성

```
┌─────────────────┐     Equinix Fabric     ┌──────────────────┐
│                 │◄─────────────────────►│                  │
│   H-Cloud       │                        │  퍼블릭 클라우드    │
│  (프라이빗)      │     저지연 상호연결       │  (AWS 등)         │
│                 │◄─────────────────────►│                  │
└─────────────────┘                        └──────────────────┘
        ▲                                          ▲
        │              ┌─────────────┐             │
        └──────────────│ Equinix DC  │─────────────┘
                       │ (서울/LA/    │
                       │  프랑크푸르트) │
                       └─────────────┘
```

에퀴닉스(Equinix)의 데이터센터와 Equinix Fabric 상호연결 기술을 통해 H-Cloud를 AWS 등 퍼블릭 클라우드 서비스와 연결하여 하이브리드 멀티클라우드 아키텍처를 구현하고 있다[6]. 이를 통해 서비스 지연을 줄이고 안정성을 높이는 효과를 달성하였다[6].

### 2.4.2 하이브리드 전략의 장점

현대오토에버의 하이브리드 클라우드 전략은 다음과 같은 장점을 제공한다[3][6][26]:

1. **워크로드 최적 배치**: 보안이 중요한 핵심 데이터와 서비스는 프라이빗 클라우드에, 탄력적 확장이 필요한 워크로드는 퍼블릭 클라우드에 배치할 수 있다.
2. **버스팅(Bursting)**: 트래픽 급증 시 퍼블릭 클라우드로 워크로드를 일시적으로 확장(Cloud Bursting)하여 대응할 수 있다.
3. **지리적 커버리지**: 에퀴닉스의 글로벌 데이터센터 네트워크를 활용하여 지리적 서비스 커버리지를 확장할 수 있다.
4. **저지연 연결**: 에퀴닉스 데이터센터가 주요 클라우드 및 네트워크 사업자와 인접한 위치에 있어, 저지연 네트워크 연결이 가능하다[6].

## 2.5 클라우드 네이티브 설계

H-Cloud는 클라우드 네이티브(Cloud Native) 원칙에 따라 설계되었다[4][5]. CNCF(Cloud Native Computing Foundation)가 정의하는 클라우드 네이티브의 핵심 요소인 컨테이너화(Containerization), 마이크로서비스, 선언적 API(Declarative APIs), 서비스 메시(Service Mesh) 등의 개념을 적극 도입하고 있다[27].

기존 애플리케이션의 클라우드 네이티브 현대화(Modernization) 과정을 거쳤으며[4], 이를 통해 다음과 같은 효과를 달성하였다:

- **개발 생산성 향상**: 독립적인 서비스 단위로 개발이 가능하여 개발 속도가 향상되었다.
- **배포 유연성**: 개별 서비스를 독립적으로 배포할 수 있어, 배포 빈도를 높일 수 있었다.
- **장애 격리**: 서비스 간 장애 전파를 방지하여 시스템 전체의 가용성이 향상되었다.
- **자원 효율성**: 컨테이너 기반 운영으로 자원 활용 효율이 높아졌다.

## 2.6 멀티 리전 아키텍처

H-Cloud는 글로벌 4개 리전에서 운영되는 멀티 리전 아키텍처를 채택하고 있다[1][7]. 각 리전은 독립적으로 운영되면서도, Rancher Prime을 통한 단일 통합 제어 인터페이스를 통해 중앙에서 관리된다[7].

멀티 리전 아키텍처의 핵심 특징은 다음과 같다:

1. **데이터 지역성(Data Locality)**: 각 리전에서 해당 지역의 데이터를 처리하여 데이터 주권(Data Sovereignty) 규정을 준수한다.
2. **장애 격리**: 특정 리전의 장애가 다른 리전에 영향을 미치지 않도록 격리되어 있다.
3. **글로벌 일관성**: 글로벌 인프라 표준에 따라 모든 리전에서 동일한 수준의 서비스를 제공한다[1].
4. **통합 관리**: Rancher Prime의 멀티 클러스터 관리 기능을 통해 모든 리전의 쿠버네티스 클러스터를 단일 콘솔에서 관리한다[7].
