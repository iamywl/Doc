# 제6장: 데브옵스(DevOps) 체계

---

## 6.1 개요

H-Cloud의 데브옵스(DevOps) 체계는 HKS의 4대 핵심 특징 중 하나로, 소프트웨어 개발(Development)과 운영(Operations)의 통합을 통해 빠르고 안정적인 서비스 배포를 가능하게 한다[4][5]. CI/CD(지속적 통합/지속적 배포) 파이프라인을 완전히 자동화하여, 개발자가 코드를 커밋하면 빌드, 테스트, 보안 검사, 이미지 빌드, 배포까지 자동으로 수행되는 체계를 구축하였다[4][5].

---

## 6.2 CI/CD 파이프라인 아키텍처

HKS의 CI/CD 파이프라인은 다음과 같은 단계로 구성된다[4][5]:

```
┌──────────┐    ┌──────────┐    ┌───────────────────┐    ┌──────────┐    ┌──────────┐    ┌─────────────────┐
│          │    │          │    │                   │    │          │    │          │    │                 │
│ 개발자    │───►│ GitLab   │───►│ SonarQube /       │───►│ Harbor   │───►│ ArgoCD   │───►│ Kubernetes      │
│ 코드 작성 │    │ 소스 관리 │    │ Black Duck        │    │ 이미지   │    │ GitOps   │    │ 클러스터         │
│          │    │          │    │ 코드 품질/보안 검사 │    │ 저장     │    │ 배포     │    │ 서비스 운영      │
└──────────┘    └──────────┘    └───────────────────┘    └──────────┘    └──────────┘    └─────────────────┘
```

### 6.2.1 소스 코드 관리 단계 (GitLab)

개발자는 GitLab에 소스 코드를 커밋한다[4][5]. GitLab은 소스 코드 버전 관리뿐만 아니라, 코드 리뷰(Merge Request), 이슈 트래킹, 프로젝트 관리 기능을 통합적으로 제공한다[31].

- **브랜치 전략**: Git Flow 또는 Trunk-Based Development 등의 브랜치 전략을 통해 코드 변경사항을 관리한다.
- **코드 리뷰**: Merge Request를 통한 동료 코드 리뷰를 수행하여 코드 품질을 보장한다.
- **파이프라인 트리거**: 코드 커밋 또는 Merge Request 생성 시 자동으로 CI 파이프라인이 트리거된다.

### 6.2.2 코드 품질 검사 단계 (SonarQube)

SonarQube를 통해 정적 코드 분석이 수행된다[4][5][34]. 이 단계에서는 다음과 같은 검사가 이루어진다:

- **버그 탐지**: 잠재적인 런타임 오류를 유발할 수 있는 코드 패턴을 탐지한다.
- **코드 스멜(Code Smell) 탐지**: 유지보수성을 저하시키는 코드 패턴을 식별한다.
- **보안 취약점 탐지**: SQL 인젝션, XSS(Cross-Site Scripting), 하드코딩된 비밀번호 등의 보안 취약점을 탐지한다.
- **코드 커버리지**: 단위 테스트의 코드 커버리지를 측정한다.
- **품질 게이트(Quality Gate)**: 사전에 정의된 품질 기준을 통과하지 못한 코드는 다음 단계로 진행되지 않는다.

### 6.2.3 오픈소스 보안 검사 단계 (Black Duck)

Black Duck을 통해 오픈소스 구성요소에 대한 보안 및 라이선스 검사가 수행된다[4][5][35]. 이 단계에서는 다음과 같은 검사가 이루어진다:

- **오픈소스 식별**: 프로젝트에 포함된 오픈소스 라이브러리와 프레임워크를 자동으로 식별한다.
- **CVE(Common Vulnerabilities and Exposures) 점검**: 식별된 오픈소스 구성요소에 알려진 보안 취약점이 있는지 점검한다.
- **라이선스 준수 확인**: 사용된 오픈소스의 라이선스 조건(GPL, Apache, MIT 등)을 확인하고, 의무사항 준수 여부를 검증한다.
- **정책 위반 탐지**: 조직의 오픈소스 사용 정책에 위반되는 구성요소를 탐지한다.

### 6.2.4 컨테이너 이미지 빌드 및 저장 단계 (Harbor)

코드 품질 및 보안 검사를 통과한 코드는 컨테이너 이미지로 빌드되어 Harbor 레지스트리에 저장된다[4][5][32]. Harbor에서는 다음과 같은 처리가 이루어진다:

- **이미지 빌드**: Dockerfile을 기반으로 컨테이너 이미지를 빌드한다.
- **이미지 스캔**: 빌드된 이미지에 대해 보안 취약점 스캔을 수행한다.
- **이미지 서명**: 신뢰할 수 있는 이미지임을 보증하기 위해 디지털 서명을 수행한다.
- **접근 제어**: 프로젝트별, 사용자별 이미지 접근 권한을 관리한다.
- **이미지 복제(Replication)**: 글로벌 리전 간 이미지 복제를 통해 모든 리전에서 동일한 이미지를 사용할 수 있도록 한다.

### 6.2.5 배포 단계 (ArgoCD)

ArgoCD를 통해 GitOps 기반의 지속적 배포가 수행된다[4][5][33]. ArgoCD는 다음과 같은 방식으로 배포를 관리한다:

- **선언적 상태 관리**: Git 저장소에 쿠버네티스 매니페스트를 통해 애플리케이션의 원하는 상태를 선언적으로 정의한다.
- **자동 동기화(Auto-Sync)**: Git 저장소의 변경사항을 감지하여 자동으로 클러스터에 반영한다.
- **상태 시각화**: 배포 상태, 리소스 상태, 동기화 상태를 웹 콘솔에서 시각적으로 확인할 수 있다.
- **롤백**: 이전 Git 커밋으로 되돌림으로써 간단하게 롤백할 수 있다.
- **멀티 클러스터 배포**: 여러 쿠버네티스 클러스터에 대한 배포를 중앙에서 관리할 수 있다.

## 6.3 GitOps 패러다임

### 6.3.1 GitOps의 정의

GitOps는 Git을 단일 진실의 원천(Single Source of Truth)으로 사용하여 인프라와 애플리케이션의 상태를 관리하는 운영 패러다임이다[33][42]. 모든 배포 변경사항은 Git 커밋을 통해 이루어지며, 이를 통해 완전한 감사 추적(Audit Trail)이 가능하다.

### 6.3.2 H-Cloud에서의 GitOps 적용

H-Cloud에서 GitOps는 다음과 같이 적용되어 있다:

- **인프라 as 코드(IaC)**: 쿠버네티스 리소스 정의(Deployment, Service, ConfigMap, Secret 등)가 Git 저장소에 코드로 관리된다.
- **환경 분리**: 개발(Dev), 스테이징(Staging), 프로덕션(Production) 환경의 설정이 별도의 Git 브랜치 또는 디렉토리로 관리된다.
- **자동 배포**: ArgoCD가 Git 저장소의 변경을 감지하여 자동으로 배포를 수행한다.
- **변경 이력 관리**: 모든 인프라 및 애플리케이션 변경사항이 Git 이력으로 기록되어, 언제 누가 무엇을 변경했는지 추적할 수 있다.

## 6.4 모니터링 및 로깅 체계

### 6.4.1 자동화된 모니터링

H-Cloud의 모니터링 체계의 가장 큰 특징은 자동화이다[4][5]. 모든 컨테이너와 가상머신의 메트릭이 자동으로 수집되며, 사용자는 별도의 설정 작업 없이 메트릭 정보를 확인할 수 있다[4].

모니터링 데이터의 종류는 다음과 같다:

- **인프라 메트릭**: CPU 사용률, 메모리 사용률, 디스크 I/O, 네트워크 트래픽 등
- **컨테이너 메트릭**: Pod 수, 컨테이너 상태, 리소스 요청/제한, 재시작 횟수 등
- **애플리케이션 메트릭**: 요청 처리량(RPS), 응답 시간, 에러율 등
- **클러스터 메트릭**: 노드 상태, 클러스터 자원 사용률, etcd 상태 등

### 6.4.2 대시보드

클러스터 상태 모니터링 대시보드가 연동되어 있어 실시간 상태 파악이 가능하다[1]. Hubble 모니터링 시스템과 Rancher 관리 콘솔의 대시보드가 연동되어, 인프라부터 애플리케이션까지 전 계층의 상태를 통합적으로 모니터링할 수 있다.

### 6.4.3 로깅

중앙 집중식 로그 수집 및 관리 체계를 운영하고 있다. 모든 컨테이너와 인프라 구성요소의 로그를 중앙으로 수집하여, 장애 분석 및 감사 목적으로 활용한다.

### 6.4.4 알림(Alerting)

사전에 정의된 임계치를 초과하거나 이상 패턴이 탐지되면 자동으로 알림이 발생한다. 알림은 운영 담당자에게 실시간으로 전달되어 신속한 대응이 가능하도록 한다.

## 6.5 DevSecOps: 보안 내재화

### 6.5.1 DevSecOps의 개념

DevSecOps는 DevOps 프로세스에 보안(Security)을 내재화한 접근 방식이다[43]. 보안을 배포 후에 점검하는 것이 아니라, 개발 단계부터 보안을 통합하여 보안 취약점을 조기에 발견하고 해결한다.

### 6.5.2 H-Cloud의 DevSecOps 적용

H-Cloud의 CI/CD 파이프라인에 SonarQube와 Black Duck을 연동하여 보안을 개발 프로세스에 내재화하였다[4][5]. 이를 통해 다음과 같은 보안 검증이 자동화되고 있다:

1. **SAST(Static Application Security Testing)**: SonarQube를 통한 정적 보안 분석으로, 소스 코드 수준에서 보안 취약점을 탐지한다[34].
2. **SCA(Software Composition Analysis)**: Black Duck을 통한 오픈소스 구성요소 분석으로, 서드파티 라이브러리의 알려진 취약점을 탐지한다[35].
3. **컨테이너 이미지 스캔**: Harbor에서 컨테이너 이미지의 보안 취약점을 스캔한다[32].
4. **라이선스 준수**: 오픈소스 라이선스 의무사항 준수 여부를 자동으로 확인한다[35].

### 6.5.3 Shift-Left 보안

H-Cloud의 DevSecOps 전략은 'Shift-Left' 원칙을 따른다. 이는 보안 검증을 소프트웨어 개발 라이프사이클의 가능한 왼쪽(초기 단계)으로 이동시키는 것을 의미한다[44]. 코드가 커밋되는 시점에서부터 보안 검사가 자동으로 수행되므로, 보안 취약점이 프로덕션 환경에 도달하기 전에 발견되고 수정된다.

## 6.6 운영 자동화

### 6.6.1 인프라 자동화

쿠버네티스의 선언적 설정(Declarative Configuration)과 자가 치유(Self-Healing) 기능을 활용하여, 인프라 운영의 상당 부분을 자동화하고 있다[29]. Pod가 비정상적으로 종료되면 쿠버네티스가 자동으로 재시작하며, 노드 장애 시에는 다른 노드로 워크로드를 자동 이전한다.

### 6.6.2 24/7 기술지원 운영

현대오토에버는 24시간 365일 모니터링 및 기술지원 운영 서비스를 제공한다[3]. 이는 커넥티드카 서비스의 무중단 운영을 보장하기 위한 필수적인 운영 체계이다.

## 6.7 DevOps 문화와 조직

DevOps는 단순히 도구의 문제가 아니라 조직 문화의 변화를 동반한다[45]. 현대오토에버는 기존의 개발과 운영이 분리된 사일로(Silo) 조직 구조에서, 개발과 운영이 협업하는 크로스펑셔널(Cross-Functional) 팀 구조로의 전환을 추진하고 있다.

이를 지원하기 위해 HKS는 개발자 친화적인 셀프서비스 기능을 제공하여, 개발자가 인프라 팀에 의존하지 않고도 컨테이너 환경을 직접 구성하고 배포할 수 있도록 한다[4].
